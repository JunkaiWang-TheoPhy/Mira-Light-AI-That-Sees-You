# CV 上展补强与验收路线

更新时间：2026-04-09

## 这份文档解决什么问题

这份说明不是在讨论“还能再做哪些炫一点的视觉玩法”，而是在回答一个更现实的问题：

> 如果 Mira Light 真的要带去展位，当前这条计算机视觉链路最应该先补什么？

结论很明确：

- 当前仓库里的 CV 主链已经做出来了
- 它已经明显超过“纯 demo 拼接”
- 但它还没有到“展位级感知稳定性”

所以当前最优先补的不是更多视觉场景，而是两件事：

1. 稳定看到人
2. 稳定跟住目标

## 当前状态判断

当前仓库已经有这条完整主链：

```text
camera / receiver
-> track_target_event_extractor.py
-> vision.latest.json / vision.events.jsonl
-> vision_runtime_bridge.py
-> MiraLightRuntime.apply_tracking_event(...)
-> lamp control
```

对应文件：

- [scripts/track_target_event_extractor.py](/Users/huhulitong/Documents/GitHub/Mira-Light/scripts/track_target_event_extractor.py)
- [scripts/vision_runtime_bridge.py](/Users/huhulitong/Documents/GitHub/Mira-Light/scripts/vision_runtime_bridge.py)
- [scripts/mira_light_runtime.py](/Users/huhulitong/Documents/GitHub/Mira-Light/scripts/mira_light_runtime.py)
- [docs/feature/20-camera-cv-runtime-bridge-progress.md](/Users/huhulitong/Documents/GitHub/Mira-Light/docs/feature/20-camera-cv-runtime-bridge-progress.md)
- [docs/mira-light-live-follow-demo-runbook.md](/Users/huhulitong/Documents/GitHub/Mira-Light/docs/mira-light-live-follow-demo-runbook.md)

当前已经具备：

- JPEG 接收与离线回放
- `target_seen / target_updated / target_lost / no_target`
- `scene_hint` 与 `control_hint`
- `target lock`、`switch margin`、短暂丢失 hold
- live tracking 写入 runtime
- touch / hand-avoid / multi-person 的第一版桥接

当前还不够强的地方：

- 主检测仍偏依赖 `haar_face + hog_person + background_motion`
- 静止的人在复杂现场仍有丢失风险
- `track_target` 的目标稳定性还不够展位级
- 目前更像“稳定跟人雏形”，不是“稳定跟任意展品”
- 导演台还缺更直接的视觉调试画面

## 一句话优先级

如果现在只能做有限工作量，顺序必须是：

1. 让它稳定看到人
2. 让它稳定跟住目标
3. 让导演台可以看见视觉状态并快速调参
4. 最后才考虑扩展更多视觉触发场景

## P0：稳定看到人

这一项是最优先级。

因为只要“人站着不动就会丢”，后面的 `wake_up`、`curious_observe`、`farewell`、`track_target` 都会一起变得不可信。

### 具体要补什么

#### 1. 升级 detector 层级

当前 extractor 的结构是：

- `haar_face`
- `hog_person`
- `background_motion`
- `skin_motion_hand`

见：

- [scripts/track_target_event_extractor.py](/Users/huhulitong/Documents/GitHub/Mira-Light/scripts/track_target_event_extractor.py#L1053)

建议调整成：

```text
更强的 person detector
-> hog_person
-> haar_face
-> background_motion fallback
```

这里的重点不是“追最新模型”，而是让 `background_motion` 彻底退回兜底角色，不再承担主展位识别职责。

#### 2. 明确 detector 的展位准入规则

当前 bridge 已经有 detector allowlist、confidence gate、persistence frames：

- [scripts/vision_runtime_bridge.py](/Users/huhulitong/Documents/GitHub/Mira-Light/scripts/vision_runtime_bridge.py#L938)

要做的不是重写，而是给它补一套明确的展位策略：

- scene start 只允许稳定 detector
- motion blob 不直接触发主场景
- 低 confidence 只记录，不触发动作
- 多人路过时优先保守，不要频繁切场景

#### 3. 建立展位回放样本集

现在仓库已经有 replay bench 和 live follow runbook：

- [scripts/vision_replay_bench.py](/Users/huhulitong/Documents/GitHub/Mira-Light/scripts/vision_replay_bench.py)
- [docs/mira-light-live-follow-demo-runbook.md](/Users/huhulitong/Documents/GitHub/Mira-Light/docs/mira-light-live-follow-demo-runbook.md)

下一步要把样本从“能回放”提升到“能验收”。

至少要覆盖这些样本：

- 单人正对灯站立不动
- 单人缓慢靠近
- 单人缓慢后退
- 背光或侧光
- 人脸部分遮挡
- 两人先后进入
- 一人站定，另一人从侧边快速经过

#### 4. 给导演台补视觉调试信息

现在导演台能看到摘要，但还不够做现场排障。

建议最少补：

- 当前 selected target
- 当前 detector 名称
- 当前 confidence
- target 是否被 lock
- 为什么触发
- 为什么没触发

如果能再往前一步，最好加：

- bbox overlay
- 当前 `scene_hint`
- 当前 `control_hint`

相关入口：

- [scripts/console_server.py](/Users/huhulitong/Documents/GitHub/Mira-Light/scripts/console_server.py)
- [web/app.js](/Users/huhulitong/Documents/GitHub/Mira-Light/web/app.js)

### P0 验收标准

这一项完成的最小标准建议写死成：

- 单人站在展位前 `5s` 不移动，不应从 `target_present=true` 掉到 `no_target`
- 单人缓慢左右移动时，不应每秒切换多次 `track_id`
- 背光下允许 confidence 下降，但不应频繁 wake / sleep 抖动
- motion-only 候选不应直接触发 `wake_up`、`curious_observe`

## P0：稳定跟住目标

这一项和“稳定看到人”一样重要，但顺序排在它后面。

因为“看得到”是“跟得住”的前提。

### 先做一个决策

必须先明确这次展位要证明的是哪一种跟随：

1. 稳定跟人
2. 稳定跟书 / 物体

当前仓库更接近第一种，也就是“稳定跟人”：

- `tracking.target_class` 主要围绕 `person`
- `selected_target` 策略是按人物目标在做
- runtime 的 `apply_tracking_event(...)` 已经能接 `control_hint`

所以如果这次展位没有硬要求“跟书”，推荐先把“稳定跟人”做透。

### 具体要补什么

#### 1. 做强 selected target continuity

当前 extractor 已经有：

- `selected_track_id`
- continuity recovery
- `switch_margin`
- `hold_missing_frames`

见：

- [scripts/track_target_event_extractor.py](/Users/huhulitong/Documents/GitHub/Mira-Light/scripts/track_target_event_extractor.py#L370)

下一步要继续做强：

- 当前主目标一旦锁定，就不要被短暂路人轻易抢走
- 当前主目标短暂遮挡后，应优先恢复同一个目标
- 多目标情况下，要优先“稳定”而不是“最优分数”

#### 2. 增加跟踪控制 deadband 和更强平滑

当前 runtime 已有平滑目标控制：

- [scripts/mira_light_runtime.py](/Users/huhulitong/Documents/GitHub/Mira-Light/scripts/mira_light_runtime.py#L1319)

但上展前建议继续补：

- 小幅画面抖动时不下发新舵机目标
- 左右误差接近 0 时直接视为保持
- pitch / yaw / lift / reach 的变化速度进一步收紧

目标不是让它“更灵”，而是让它“更稳”。

#### 3. 收紧 tracking update 的节奏

当前 bridge 已经有限频与 persistence：

- [scripts/vision_runtime_bridge.py](/Users/huhulitong/Documents/GitHub/Mira-Light/scripts/vision_runtime_bridge.py#L1075)

上展前应该把这层当成正式调参面，而不是临时参数：

- `tracking-update-ms`
- `tracking-persistence-frames`
- `tracking-min-confidence`

展位里，稍慢一点但稳，比高速乱跳强得多。

#### 4. 如果主秀一定要跟书，就单独补 object detector

如果这次 demo 明确要演“书往左中右移动，灯稳定跟书”，那不要继续只调 person detector。

这时应该直接新增：

- 书本 / 指定目标物 detector
- `target_class = object` 的单独策略
- object 版 `scene_hint / control_hint`

否则会出现：

- 人跟随已经差不多
- 但书本跟随一直若有若无

这会直接影响现场可信度。

### P0 验收标准

- 单人左右移动时，灯头应连续跟随，不出现明显抽搐
- 同一目标缓慢移动时，`track_id` 不应频繁重置
- 两人同时出现时，没有显式锁定的情况下，系统应偏保守而不是来回跳
- 目标消失后，tracking 应清空并进入 `farewell / sleep` 的有序过渡

## P1：导演台必须能调，不能只会看

如果要在展位上真的跑，这一层很关键。

建议导演台增加这些最小工具：

- 锁定当前目标
- 解除目标锁定
- 一键清 tracking 状态
- 显示最近几条 vision 事件
- 显示当前 gate reason
- 一键复制当前 runtime / vision state

这部分不是锦上添花，而是现场排障必须品。

## 推荐实施顺序

建议按下面顺序推进，不要反过来：

### 第一步：先固化验收样本

先做离线 replay 样本集和命名规范，再调 detector。

原因：

- 没有固定样本，任何调参都不可复现
- 没有固定回放，现场感觉很容易误导判断

### 第二步：补强 person detection

把“稳定看到人”做到可复现通过。

这一步不要顺手扩需求，不要一边调人、一边调书、一边调手势。

### 第三步：补强 selected target continuity

把目标保持、切换阈值、短时丢失恢复做稳。

### 第四步：补强 tracking smoothing

在 runtime 里把跟踪从“能动”调到“耐看”。

### 第五步：补导演台调试视图

把调参和排障能力交给现场操作人。

### 第六步：做真机长时间跑测

建议至少做：

- `30` 分钟连续运行
- 单人场景
- 多人穿插场景
- 手靠近但不触发误躲避
- 反复出现 / 离开

## 上展前最小检查表

在正式上展前，至少确认这些问题都能回答：

1. 单人静止时会不会丢？
2. 双人同时出现时会不会乱跳？
3. 目标短暂被挡住会不会立刻丢？
4. tracking 时灯会不会一步一跳？
5. 视觉误判时，导演台能不能一眼看到原因？
6. 当前展位 demo 到底是跟人，还是跟书？

如果这 6 个问题还答不清楚，就不建议把重点放在新视觉场景上。

## 最后的原则

当前阶段最重要的不是：

- 更多视觉彩蛋
- 更多自动触发
- 更复杂的多模态叙事

而是先把这一条闭环打磨到可信：

```text
稳定 detector
-> 稳定 selected target
-> 稳定 control hint
-> 稳定 runtime tracking
```

只有这条链稳定了，`wake_up / curious_observe / track_target / farewell / sleep` 才真正能像一个展位系统，而不是一组偶尔成功的 demo 片段。

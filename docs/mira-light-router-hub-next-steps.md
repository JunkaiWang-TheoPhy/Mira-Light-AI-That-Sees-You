# Mira Light 路由器中转枢纽下一步执行清单

## 文档目的

这份文档把当前方案拆成下一步真正要做的事情，避免停留在架构讨论。

## 当前工作目标

当前阶段的目标不是“一步到位做完整系统”，而是尽快拿到第一条可验证通路：

```text
OpenClaw -> 插件 -> 服务器本地 bridge 地址 -> 隧道 -> 本地中转枢纽 -> ESP32
```

## 推荐推进顺序

### 第 1 优先级：先确认本地硬件和网络事实

必须先拿到以下信息：

- `ESP32` 当前 IP
- `ESP32` 是否能固定 IP
- 本地中转枢纽具体是哪台设备
- 该设备是否能长期在线
- 该设备是否能装 `autossh`
- 该设备是否能跑 bridge 服务

### 第 2 优先级：做本地最小 bridge

第一版 bridge 只需要做到：

- 状态转发
- 灯光转发
- 动作转发
- 舵机控制转发
- 健康检查

这一版先不要追求复杂场景脚本。

### 第 3 优先级：打通第一条反向隧道

关键验收项是：

- 服务器可以 `curl http://127.0.0.1:9783/health`
- 服务器可以 `curl http://127.0.0.1:9783/v1/mira-light/status`

### 第 4 优先级：接入服务器端 OpenClaw 插件

第一版插件只需覆盖：

- 读取状态
- 读取灯光
- 设置灯光
- 执行动作
- 停止动作
- 控制舵机
- 复位

### 第 5 优先级：做最小端到端验收

建议最小验收用例：

1. 读取状态成功
2. 切换暖白灯成功
3. 执行 `wave` 成功
4. 执行 `dance` 成功
5. 单关节控制成功

## 建议拆成的交付项

### 交付项 A：本地网络与设备确认单

至少应包含：

- `ESP32` 的固定 IP
- 路由器 DHCP 保留截图或记录
- 本地中转枢纽型号
- 本地中转枢纽操作系统

### 交付项 B：本地 bridge 交付物

至少应包含：

- 服务源码
- 环境变量说明
- 启动方式
- 健康检查接口
- token 配置方式

### 交付项 C：隧道交付物

至少应包含：

- `autossh` 启动命令
- 守护方式
- 自启动配置
- 断线重连策略

### 交付项 D：OpenClaw 插件交付物

至少应包含：

- 插件目录
- 配置 schema
- 工具定义
- bridge 调用逻辑
- 错误处理逻辑

### 交付项 E：联调验收记录

至少应包含：

- 本地访问 ESP32 成功记录
- 本地访问 bridge 成功记录
- 服务器访问 tunnel 后 bridge 成功记录
- OpenClaw 调工具成功记录

## 当前建议的实施分期

### Phase 0：确认条件

目标：

- 弄清设备、网络和中转硬件

完成标志：

- 可以在本地中转枢纽上成功 `curl http://ESP32_IP/status`

### Phase 1：打通 bridge

目标：

- 在本地设备上跑通 `mira-light bridge`

完成标志：

- `curl http://127.0.0.1:9783/health` 成功
- `curl http://127.0.0.1:9783/v1/mira-light/status` 成功

### Phase 2：打通云端访问

目标：

- 让服务器能通过隧道访问 bridge

完成标志：

- 服务器本地 `127.0.0.1:9783` 返回桥接数据

### Phase 3：接入 OpenClaw

目标：

- 插件调用 bridge

完成标志：

- OpenClaw 可调用 `mira_light_get_status`
- OpenClaw 可调用 `mira_light_set_led`
- OpenClaw 可调用 `mira_light_run_action`

### Phase 4：升级为长期运行

目标：

- 消除“手工临时链路”

完成标志：

- bridge 自启动
- 隧道自启动
- 断线自动恢复
- token 和日志都已固化

## 推荐责任分工

### 硬件/网络侧

负责：

- 固定 `ESP32` 地址
- 确认局域网连通性
- 选择本地中转枢纽

### bridge 开发侧

负责：

- 实现本地 HTTP bridge
- 增加鉴权、超时和健康检查

### 云端/OpenClaw 侧

负责：

- 接入 `mira-light` 插件
- 配置 `bridgeBaseUrl`
- 做工具级联调

### 验证侧

负责：

- 按阶段验收链路
- 记录成功命令与失败原因

## 推荐的近期执行清单

建议下一步直接按这张清单推进：

1. 确认哪台设备充当本地中转枢纽
2. 给 `ESP32` 固定 IP
3. 在本地中转枢纽上验证 `ESP32` API
4. 实现第一版 `mira-light bridge`
5. 用 `ssh -R` 先手工打通隧道
6. 在服务器上 `curl` bridge 地址
7. 接入最小版 `mira-light` 插件
8. 完成一次 `wave` 和 `solid` 的端到端控制
9. 再把 bridge 和隧道改造成常驻服务

## 一句话总结

下一步不是继续抽象讨论“能不能接”，而是用最小四件套把第一条真实链路做出来：

- 固定 IP
- 本地 bridge
- 反向隧道
- OpenClaw 插件

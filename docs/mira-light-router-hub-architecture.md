# Mira Light 路由器中转枢纽架构说明

## 文档目的

这份文档解释：

- 为什么 `Mira Light` 不应该由云服务器直接访问
- 路由器中转枢纽在整体架构中的位置
- 推荐的部署形态是什么
- 每一层分别负责什么

## 先说结论

当 `OpenClaw` 运行在远端云服务器上，而 `ESP32 Mira Light` 运行在本地局域网中时，推荐架构不是：

```text
OpenClaw -> 直接访问 ESP32
```

而是：

```text
OpenClaw
-> mira-light 插件
-> 服务器本地 bridge 地址
-> 反向隧道 / 私网通道
-> 本地中转枢纽
-> ESP32 Mira Light
```

当前已知背景中，服务器上的 `OpenClaw` 运行在本地模式，并绑定在：

```text
127.0.0.1:20656
```

这意味着整个接入设计天然更适合走“服务器回环地址 -> 本地隧道 -> 本地中转枢纽”的模式，而不是走公网暴露式设计。

## 为什么不推荐服务器直接访问 ESP32

### 1. ESP32 通常在私有局域网

`ESP32` 的地址通常长这样：

- `192.168.31.42`
- `192.168.0.101`
- `10.0.0.25`

这类地址只能在本地网络中访问。  
云服务器通常看不到这些地址。

### 2. 直接公网暴露风险过高

如果要让云服务器直连 ESP32，常见做法通常是：

- 家里路由器做端口映射
- 把 ESP32 直接开放到公网
- 给单片机挂一个公网隧道

这几种方式都不适合作为第一阶段方案，因为它们会带来：

- 攻击面过大
- 鉴权边界弱
- 后续难以审计
- 设备端容易背负过多安全职责

### 3. OpenClaw 更适合调用“受控工具”

`OpenClaw` 更适合调用明确、可控、可鉴权的工具层，而不是直接操作一个裸露的硬件地址。

也就是说，`OpenClaw` 的合理上游对象应该是：

- 插件
- bridge 服务
- 受控 API

而不是：

- 家里路由器的公网映射端口
- 裸露在公网的 `ESP32`

## 路由器中转枢纽到底是什么

这里说的“路由器”并不是单指普通家用 Wi‑Fi 盒子，而是指一台放在本地网络里、长期在线、能承担转发和服务职责的边缘节点。

它至少需要具备以下能力：

- 能访问本地 `ESP32`
- 能运行一个轻量 `HTTP bridge`
- 能建立到云服务器的长期连接
- 能在断线后自动恢复

典型形态包括：

### 形态 A：OpenWrt 路由器直接承担中转职责

适合条件：

- 路由器可安装软件包
- 有足够存储和运行空间
- 能跑 `autossh`、`curl`、脚本服务

优点：

- 常在线
- 和 ESP32 在同一网段时路径短
- 不依赖笔记本电脑

限制：

- 算力和开发环境可能较弱
- 调试体验不如小主机

### 形态 B：普通路由器 + 边缘主机

例如：

- 家用路由器 + 树莓派
- 家用路由器 + mini PC
- 家用路由器 + NAS 容器

这是最推荐的长期方案。

优点：

- 路由器继续做网络入口
- 边缘主机做 bridge、日志、隧道和守护进程
- 可维护性最好

### 形态 C：临时电脑充当中转枢纽

这是最快验证链路的方式，但不适合作为长期方案。

优点：

- 最快开始
- 不需要额外设备

限制：

- 容易睡眠
- 网络不稳定
- 不适合长期在线

## 推荐职责拆分

### ESP32 Mira Light

负责：

- 执行底层硬件动作
- 提供现有 REST API

不负责：

- 对公网暴露
- 长连接守护
- 复杂鉴权策略

### 本地中转枢纽

负责：

- 访问 `ESP32` 局域网地址
- 对外提供受控 `bridge`
- 持续连接云服务器
- 必要时做请求鉴权、限速、审计

### 云服务器

负责：

- 运行 `OpenClaw`
- 装载 `mira-light` 插件
- 调用 bridge，而不是直接访问单片机

### OpenClaw 插件

负责：

- 把灯光、动作、舵机控制包装成工具
- 做参数约束和错误处理
- 用面向能力的方式暴露给模型

## 推荐链路

最推荐的第一阶段链路如下：

```text
用户
-> OpenClaw
-> mira-light 插件
-> http://127.0.0.1:9783
-> SSH 反向隧道
-> 本地中转枢纽 127.0.0.1:9783
-> http://ESP32_IP
-> ESP32 执行动作
```

这里有两个关键点：

### 1. 服务器只访问本机地址

插件只看到：

```text
http://127.0.0.1:9783
```

这很好，因为：

- 不需要把本地设备直接挂公网
- 与当前 `OpenClaw` 本地模式更匹配
- 安全边界更清晰

### 2. 本地中转枢纽只访问 ESP32 内网地址

例如：

```text
http://192.168.31.42
```

这样 `ESP32` 永远只待在局域网里。

## 为什么反向隧道最适合当前阶段

### 1. 云服务器不需要入侵本地网络

如果使用反向隧道，连接是本地主动发起到云端，而不是云端主动打进你家网络。

这意味着：

- 不需要家里端口映射
- 不需要给运营商公网 IP
- 不需要公网开放设备端口

### 2. 与 OpenClaw 的 loopback 模式天然兼容

如果服务器上的 `OpenClaw` 是本地模式，只绑定在 `127.0.0.1`，那么 bridge 映射到服务器本机回环地址，正好最合适。

### 3. 后续可以平滑升级

今天可以先用：

- `SSH -R`
- `autossh`

未来如果想长期化，可以升级为：

- `Tailscale`
- `WireGuard`
- 专门的私网 service mesh

但即使升级网络层，bridge 这层仍然应该保留。

## bridge 为什么不能省

有时候会有人想直接把隧道终点指向 ESP32。  
这不推荐，因为 bridge 不是多余层，而是整个系统的安全与维护边界。

bridge 至少解决了这些问题：

- API 统一入口
- 请求鉴权
- 参数校验
- 错误信息整形
- 日志记录
- 后续接入多设备时的扩展

例如：

- 今天只转发 `Mira Light`
- 明天可以同时桥接摄像头、传感器、麦克风

## 推荐的最小 bridge API

本地 bridge 对 OpenClaw 暴露的入口建议是：

- `GET /v1/mira-light/status`
- `GET /v1/mira-light/led`
- `POST /v1/mira-light/led`
- `POST /v1/mira-light/action`
- `POST /v1/mira-light/action/stop`
- `POST /v1/mira-light/control`
- `POST /v1/mira-light/reset`

这样做的好处是：

- 与 ESP32 现有接口能力一一对应
- 路由清晰
- 后续 OpenClaw 插件最容易实现

## 信任边界建议

推荐把边界切成四层：

### 第 1 层：ESP32

只相信来自本地 bridge 的请求。

### 第 2 层：本地 bridge

负责：

- 校验 `bridgeToken`
- 校验 JSON 结构
- 限制非法动作范围

### 第 3 层：云服务器上的插件

负责：

- 语义工具封装
- 高层参数限制
- 错误重试和超时

### 第 4 层：OpenClaw

负责：

- 理解用户意图
- 选择工具
- 不直接接触内网设备地址

## 最推荐的长期形态

对于长期运行，推荐优先级如下：

1. 家用路由器 + 树莓派/mini PC 作为本地中转枢纽
2. `OpenWrt` 路由器直接跑 bridge 与隧道
3. 临时笔记本电脑桥接，仅用于验证

## 一句话总结

路由器中转枢纽方案的本质，不是“把路由器变成公网入口”，而是“把它变成离硬件最近、长期在线、受控的边缘 bridge 节点”，再由它把 `Mira Light` 安全接入远端 `OpenClaw`。

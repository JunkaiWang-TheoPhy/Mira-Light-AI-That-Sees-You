## Current Blockers

Latest observed state:

- `health_connect_ready = false`
- `local_source_ready = false`
- `metrics_ready = false`
- `bluetooth_ready = true`
- `health_connect_status = missing_permissions`
- `xiaomi_provider_status = no_provider_deploy`
- `xiaomi_log_status = inaccessible:IllegalStateException`
- `band_status = bonded`
- `android.permission.BLUETOOTH_CONNECT = granted=true`
- `android.permission.POST_NOTIFICATIONS = granted=true`
- the phone now has the official `com.google.android.apps.healthdata` provider package installed

This means the Android gateway transport, Bluetooth status polling, and desktop bridge are working. The current blocker is now narrower:

- the gateway now probes Xiaomi-local sources before Health Connect
- the Xiaomi provider probe runs inside the app and currently reports guessed paths as `no provider deploy`
- the Xiaomi log fallback parser exists and is tested, but the Android app cannot currently read `com.mi.health` log files from its own sandbox on this phone
- so heart rate, SpO2, and steps remain `null` even though the band is bonded, the gateway is live, and local-source diagnostics are now exposed over HTTP

The new desktop `adb` bridge bypasses this blocker because it reads Xiaomi Fitness evidence from the Mac side. The latest verified desktop bridge snapshot on `2026-03-15 09:45 +08:00` returned:

- `heart_rate_bpm = 83`
- `spo2_percent = 97`
- `steps = 2855`
- `distance_m = 1801`
- `calories_kcal = 104`
- `connection.status = connected`

The latest transport verification also showed:

- `phone.adb_target = 4722a997`
- `phone.adb_transport = usb`
- `wireless_adb.enabled = false`

The gateway app now exposes richer runtime diagnostics:

- `/status` includes `local_source_ready` and `metrics_ready`
- `/debug/source` includes:
  - `xiaomi_provider_status`
  - `xiaomi_provider_probe`
  - `xiaomi_log_status`

The desktop bridge now exposes:

- `/v1/band/status`
- `/v1/band/latest`
- `/v1/band/events`
- `/v1/band/alerts`
- `/v1/band/debug/evidence`

The desktop bridge now resolves its ADB target in this order:

1. `OPENCLAW_MI_BAND_ADB_TARGET`
2. `wireless_adb.enabled + wireless_adb.host:port`
3. fallback USB serial `4722a997`

That means wireless ADB is now available for later use, but it is not active in the current verified setup.

## Manual Steps Still Required On The Phone

1. Keep `Mi Band Gateway` in foreground when starting the gateway.
2. In `Mi Band Gateway`, tap:
   - `Grant Android Permissions`
   - `Open Health Connect`
   - `Grant Health Permissions`
   - `Start Gateway`
3. In `Health Connect`, grant the gateway read access for heart rate, SpO2, and steps.
4. In Xiaomi Fitness, allow sync/write access into Health Connect if that option appears.

## Expected Next-State Signal

If either Xiaomi-local provider paths become readable or Xiaomi Fitness writes into granted Health Connect successfully, these values should change:

- `health_connect_ready` should become `true`
- `local_source_ready` may become `true`

- the phone stays `adb` authorized
- Xiaomi Fitness continues logging current metrics
- the band remains paired to the same phone

# Mira Light 设备连接排障说明

## 文档目的

这份文档专门回答：

> 为什么控制台现在还显示没有接入？

重点不是架构设计，而是排查真实设备联通问题。

## 当前最新已确认事实

截至当前这次检查，本机网络状态如下：

- 本机当前 IP：`192.168.0.101`
- 当前控制台目标地址曾配置为：`http://172.20.10.3`

并且当前实测：

```bash
curl http://172.20.10.3/status
```

超时失败。

这说明：

> 当前没有证据证明 `172.20.10.3` 上有一盏可响应的 `Mira Light`。

## 为什么控制台会显示未接入

网页导演台并不是“假装设备在线”。

它会尝试真正请求：

- `/status`
- `/led`
- `/actions`

如果这些请求失败，就会表现成：

- `DEVICE = UNKNOWN`
- `LAST ERROR = Failed to reach Mira Light ...`

所以当控制台显示未接入时，首先应该理解为：

> 底层设备地址不可达，而不是页面本身有 bug。

## 最常见的 5 个原因

### 1. 设备根本没开机

这是最常见原因。

### 2. 设备连的不是当前这张网

如果电脑在：

```text
192.168.0.x
```

而灯还在：

```text
172.20.10.x
```

那它们就不是当前同一网络平面。

### 3. `172.20.10.3` 是旧地址

这个值可能只是你之前记下来的旧 IP，不一定还是当前真实地址。

### 4. 设备虽然联网了，但 HTTP 服务没起来

也就是说：

- ESP32 连上了 Wi‑Fi
- 但程序没有成功启动 Web server

### 5. 控制台地址填错

控制台里写的是：

```text
http://172.20.10.3
```

但真实设备地址可能已经变成了：

```text
http://192.168.0.xxx
```

## 最重要的排查顺序

一定要按下面顺序查，不要一上来就怀疑 OpenClaw、bridge 或网页导演台。

### 第 1 步：先确认设备当前真实 IP

这一步最重要。

你可以通过以下方式确认：

#### 方式 A：看串口日志

如果设备接在这台电脑上，打开串口日志，找类似：

```text
WiFi connected
IP address: 192.168.0.123
```

#### 方式 B：看路由器或热点客户端列表

去当前使用的热点 / 路由器后台里，看有没有：

- `ESP32`
- `Mira Light`
- 未知设备但 MAC 对得上

#### 方式 C：如果你知道它在当前子网里，逐个试

例如当前电脑在：

```text
192.168.0.101
```

那更合理的是优先在：

```text
192.168.0.x
```

而不是继续假设 `172.20.10.3`。

## 第 2 步：直接测试设备原始 API

一旦拿到真实 IP，先不要通过导演台，直接跑：

```bash
curl http://<真实灯IP>/status
curl http://<真实灯IP>/led
curl http://<真实灯IP>/actions
```

如果这些请求成功，说明：

- 设备在线
- 网络打通
- Web server 正常

只有这一步成功，后面才值得继续看控制台、scene runner、OpenClaw。

## 第 3 步：再看导演台

确认原始 API 可达之后，把导演台里的 `Lamp Base URL` 改成真实 IP。

也就是说，控制台里的值必须来自真实验证，而不是猜测。

## 快速判断表

| 现象 | 更可能的问题 |
| --- | --- |
| `curl /status` 超时 | 设备不在线 / IP 不对 / 不在同一网络 |
| `ping` 不通，但本机网络正常 | 大概率 IP 不对或设备没联网 |
| `ping` 通但 `curl /status` 不通 | Web server 没起来 |
| 控制台 `DEVICE=UNKNOWN` 且 `LAST ERROR` 是 reach failed | 控制台本身没问题，是底层地址不可达 |
| 控制台选场景没反应 | scene runner 没机会真正请求到灯 |

## 当前最合理的下一步

基于当前最新事实：

- 电脑现在在 `192.168.0.101`
- 目标还是 `172.20.10.3`
- 请求超时

当前最合理结论是：

> 先不要继续假设 `172.20.10.3` 正确，而应该优先重新确认灯当前真实 IP。

## 一句话总结

现在“没接入”的原因，最可能不是控制台本身，而是：

> 当前并没有一盏在线设备在 `http://172.20.10.3` 上响应。

正确排障顺序是：

```text
先找真实 IP
-> 再 curl 原始 API
-> 再改控制台 Base URL
-> 最后再谈 OpenClaw / bridge / 云端接入
```

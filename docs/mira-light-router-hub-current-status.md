# Mira Light 路由器中转枢纽当前状态说明

## 文档目的

这份文档用于明确当前已经确认的事实、当前尚未完成的部分，以及为什么现阶段推荐走“本地中转枢纽 + bridge + 隧道”的接入路线。

## 当前状态一览

### 已确认

#### 1. ESP32 端已经具备可调用 REST API

根据 [`danpianji.html`](../docs/danpianji.html) 与已有整理文档，目前已经明确 `Mira Light` 支持：

- `GET /status`
- `POST /control`
- `POST /reset`
- `GET /led`
- `POST /led`
- `GET /actions`
- `POST /action`
- `POST /action/stop`

也就是说，设备侧已经不是“只能烧录固件的黑盒”，而是一台可以被 HTTP 调用的网络设备。

#### 2. 当前仓库已经整理出设备交付和 OpenClaw 接入思路

仓库里已经有两份关键基础文档：

- [`esp32-smart-lamp-delivery-spec.md`](../docs/esp32-smart-lamp-delivery-spec.md)
- [`openclaw-esp32-control-guide.md`](../docs/openclaw-esp32-control-guide.md)

这意味着：

- 设备接口定义已经基本清楚
- OpenClaw 侧应该走插件/bridge 路线，这一点方向已经明确

#### 3. 云端 OpenClaw 与本地 ESP32 不在同一网络平面

根据当前交付背景，`OpenClaw` 在远端服务器上，而 `ESP32` 大概率在你本地局域网中。

并且当前已知服务器上的 `OpenClaw` 运行在本地模式，网关绑定在：

```text
127.0.0.1:20656
```

这意味着云服务器通常不能直接访问：

- `192.168.x.x`
- `10.x.x.x`
- `172.16.x.x` 到 `172.31.x.x`

所以“服务器直接访问 ESP32”不是默认可行路径。

同时也意味着：

- 插件更适合访问服务器本机回环地址
- bridge 更适合通过隧道映射到 `127.0.0.1`
- 整个设计应该尽量避免新增公网暴露面

#### 4. 当前推荐方案已经明确

当前最稳妥方案已经不是直连，而是：

```text
ESP32
-> 本地中转枢纽
-> 本地 bridge
-> 反向隧道 / 私网通道
-> 云服务器
-> OpenClaw 插件
-> OpenClaw
```

### 尚未完成

#### 1. 还没有一台正式启用的本地中转枢纽

虽然“路由器作为中转枢纽”的方向已经明确，但当前还没有看到以下之一已经正式落地：

- 可编程 `OpenWrt` 路由器
- 软路由
- 树莓派
- 小主机/NAS 容器

#### 2. 还没有正式的 `mira-light bridge`

当前仓库说明了 bridge 应该做什么，但还没有实际 bridge 服务交付物，例如：

- 可运行服务
- 守护脚本
- 配置文件
- 鉴权令牌
- 日志路径

#### 3. 还没有服务器侧 `mira-light` 插件交付

当前还没有看到正式的：

- `mira-light` 插件目录
- 插件配置文件
- 已注册工具
- 集成测试结果

#### 4. 还没有完成端到端链路验证

当前还缺少以下验证证据：

- 本地中转枢纽能访问 `ESP32`
- 反向隧道稳定建立
- 服务器本地 `127.0.0.1:9783` 能拿到设备状态
- `OpenClaw` 能成功调用灯光、动作、舵机控制工具

## 当前缺口清单

从“已有设备 API”走到“OpenClaw 可稳定控制”之间，至少还差以下交付：

### 缺口 1：设备网络信息还未固化

还需要明确：

- ESP32 当前 IP
- 是否已经固定 IP
- 是否已经做 DHCP 保留
- 是否长期接在稳定 `2.4GHz` Wi‑Fi

### 缺口 2：中转枢纽硬件未选型

还需要确定到底由谁来承担本地 bridge：

- 路由器自己
- 路由器后的树莓派
- 路由器后的小主机
- 临时笔记本

### 缺口 3：bridge API 还未真正实现

虽然路由设计已经清楚，但还没有把它落成实际运行的服务。

### 缺口 4：安全边界还未落地

还需要明确：

- bridge 是否要求 `token`
- 服务器侧是否只访问 `127.0.0.1`
- 隧道如何自动重连
- 本地服务如何开机自启

## 当前风险

### 风险 1：把“路由器”理解成普通家用路由器

如果只是普通不可编程路由器，那么它无法承担：

- bridge 运行
- 隧道守护
- 自动重连

这时需要额外一台边缘主机。

### 风险 2：直接做公网暴露

如果为了省事直接：

- 开家庭端口映射
- 暴露 ESP32
- 暴露 bridge 公网地址

会大幅增加安全风险，不适合作为第一阶段方案。

### 风险 3：用笔记本做长期中转

笔记本适合验证，不适合长期运行，因为会遇到：

- 休眠
- 换网络
- 合盖
- 电池策略

### 风险 4：跳过 bridge 直接隧道到 ESP32

这样虽然看起来少一层，但会损失：

- 参数校验
- 工具边界
- 鉴权
- 日志
- 可扩展性

## 当前推荐判断

如果现在就要推进实施，推荐的现实判断是：

### 第一阶段目标

先验证链路，不追求完美部署。

最小验证路径：

- 一台本地中转机
- 一个最小 bridge
- 一条反向隧道
- 一个服务器侧最小插件

### 第二阶段目标

把“临时链路”升级为“长期在线链路”。

升级项包括：

- 固定 IP
- `autossh` 或等价守护
- 开机自启
- `bridgeToken`
- 日志与健康检查

### 第三阶段目标

从“能控制”升级到“可编排场景”。

这时再接入：

- 自然语言动作映射
- 动作原语库
- 场景脚本
- 更高层的 `wake_up`、`curious`、`farewell` 等状态

## 当前最应该做的事

当前不是继续讨论抽象架构，而是尽快把下面几个事实确定下来：

1. `ESP32` 的固定 IP 是多少
2. 本地中转枢纽到底是哪台设备
3. bridge 准备用什么语言和运行方式实现
4. 服务器侧 `mira-light` 插件放在哪个 OpenClaw 环境里
5. 第一条反向隧道是否能稳定打通

## 一句话总结

当前项目已经完成了“设备能力定义”和“推荐接入路线”的前半段，但还没有进入真正的 bridge、隧道、插件、联调交付阶段。下一步的重点应该从“讲方案”切到“做第一条通路”。

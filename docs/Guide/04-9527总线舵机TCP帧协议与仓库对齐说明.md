# 9527 总线舵机 TCP 帧协议与仓库对齐说明

更新时间：2026-04-09

## 结论

`192.168.31.10:9527` 是 Mira Light 当前联调使用的原始 TCP 总线舵机控制口。

本仓库现已统一按 TCP 协议对齐，默认目标写为：

```text
tcp://192.168.31.10:9527
```

同时，实际下发到设备的原始帧必须严格遵循以下格式：

- 单舵机：`#003P1500T1000!`
- 多舵机：`{#001P2000T1000!#003P0833T2000!}`

## 帧格式规则

### 单舵机控制帧

格式：

```text
#000P1500T1000!
```

字段解释：

- `#` 和 `!` 为固定英文格式符号
- `000` 为舵机 `ID`
- `P1500` 为 PWM 脉宽
- `T1000` 为时间，单位毫秒

严格要求：

- `ID` 范围 `000-254`，必须始终为 3 位，不足补 `0`
- `PWM` 范围 `0500-2500`，必须始终为 4 位，不足补 `0`
- `TIME` 范围 `0000-9999`，必须始终为 4 位，不足补 `0`

例如：

- `3` 号舵机必须写成 `003`
- `PWM 800` 必须写成 `P0800`
- `TIME 500ms` 必须写成 `T0500`

### 多舵机控制帧

格式：

```text
{#000P1602T1000!#001P2500T0000!#002P1500T1000!}
```

规则：

- 当 2 个或以上舵机指令同时下发时，整条命令必须用 `{}` 包住
- 花括号里面是若干条合法的单舵机帧直接拼接
- 不能在内部插入额外字符、空格或非法分隔符

## 典型控制示例

### 示例 1：控制一个 270 度总线舵机

舵机 `ID=003`，居中位置：

```bash
printf '#003P1500T1000!\n' | nc 192.168.31.10 9527
```

逆时针最快转动 135 度：

```bash
printf '#003P2500T0000!\n' | nc 192.168.31.10 9527
```

顺时针最慢转动 270 度：

```bash
printf '#003P0500T9999!\n' | nc 192.168.31.10 9527
```

说明：

- 舵机轴朝向自己时，`1500 -> 2500` 表示逆时针方向
- `1500 -> 0500` 表示顺时针方向
- `0500-2500` 这一千档位对应 270 度范围
- `T0000-T9999` 表示移动时间，单位毫秒

### 示例 2：同时控制两个 270 度总线舵机

舵机 `ID=001` 和 `ID=003` 的中位控制：

```bash
printf '{#001P1500T1000!#003P1500T1000!}\n' | nc 192.168.31.10 9527
```

若：

- `001` 号舵机逆时针用 `1000ms` 转动 `67.5°`
- `003` 号舵机顺时针用 `2000ms` 转动 `90°`

则命令为：

```bash
printf '{#001P2000T1000!#003P0833T2000!}\n' | nc 192.168.31.10 9527
```

说明：

- `#001P2000T1000!`
  `2000` 是 `1500-2500` 中间值，等于 135 度的一半，即 `67.5°`
- `#003P0833T2000!`
  `1500-0500` 的一千档位对应 135 度，因此 `90°` 约为：

```text
1500 - ((90 / 135) * 1000) ≈ 833
```

## 本仓库的协议对齐方式

当前仓库已经改成：

- 导演台和 bridge 对外仍保留 HTTP 工作流
- 真正对灯发送时，统一转换为 raw TCP 舵机帧
- 不再向 `9527` 发送 `GET /status`、`GET /led`、`GET /actions`

也就是说，当前仓库的架构是：

```text
Director Console / Runtime HTTP
        ->
Local Bridge
        ->
Raw TCP Servo Frames
        ->
192.168.31.10:9527
```

### 2. 帧校验

文件：

- [scripts/bus_servo_protocol.py](/Users/huhulitong/Documents/GitHub/Mira-Light/scripts/bus_servo_protocol.py)

### 4. 逻辑角度到 PWM 的映射

文件：

- [scripts/bus_servo_mapping.py](/Users/huhulitong/Documents/GitHub/Mira-Light/scripts/bus_servo_mapping.py)

职责：

- 把 `servo1-4` 的逻辑角度换算成 PWM
- 例如当前映射：

  `servo2 -> 001`
  `servo3 -> 002`
  `servo4 -> 003`

### 5. Runtime TCP 主链

文件：

- [scripts/mira_light_runtime.py](/Users/huhulitong/Documents/GitHub/Mira-Light/scripts/mira_light_runtime.py)

职责：

- `status` / `led` / `actions` 由 bridge 侧缓存或模拟返回

## 当前建议

后续若继续扩展 Mira Light 的 TCP 控制链，建议始终遵循：

1. 所有 raw TCP 下发统一走 [scripts/bus_servo_protocol.py](/Users/huhulitong/Documents/GitHub/Mira-Light/scripts/bus_servo_protocol.py)
2. 不要手写不经校验的帧字符串直接发设备
3. 导演台、bridge、scene 继续保留高层控制语义
4. 设备侧始终只接收经过格式化和校验后的 TCP 舵机帧

这样可以同时保证：

- 现场稳定性
- 协议一致性
- 场景层与底层控制解耦
- 后续排障时可直接比对原始帧
